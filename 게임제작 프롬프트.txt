**1. 확정된 요구사항 요약**
- 엔진: `Unity 6.3`
- 렌더링: `URP`
- 시점: `플레이어를 따라다니는 고정 각도의 쿼터뷰 카메라`
- 입력: `New Input System`
- 플랫폼: `PC + Android`, 같은 씬에서 동작
- 이동: `CharacterController`, `점프 없음`, `평지 전용`
- 맵: `작은 단일 맵`, `완전 수동 배치형`
- 아트 방향: `무료 에셋 전제`
- 핵심 생존 스탯: `체온 100`, `체력 100`
- 체온 규칙: 기본 `초당 -1`, 눈보라 강도에 따라 추가 감소, 화톳불 범위 안에서 `초당 +5`
- 체력 규칙: 체온이 `0`이면 `초당 -2`
- 화톳불: 총 `3개`, 시작 시 `메인 캠프 1개만 점등`, 나머지 2개는 꺼짐
- 화톳불 상호작용: 플레이어가 근처에 가면 자동 처리, 꺼진 화톳불은 인벤토리 장작 `1개` 자동 소모 후 점화
- 연료 규칙: 장작 `1개당 3초` 유지
- 자원 루프: 월드에는 `장작 오브젝트만 배치`, 장작 `5개`로 도구 `1개` 제작, 도구는 `영구 보유`
- 음식 루프: 도구 보유 후 `화톳불 근처에서만` 장작 `5개` 소모로 음식 `1개` 자동 생성
- 음식 효과: 음식 `1개` 획득 시 체력 `+1`, 체력이 가득 차 있으면 대신 돈 `+10`
- HUD: `체온 바`, `체력 바`, `장작 수`, `도구 수/보유 여부`, `음식 수`, `돈`, `생존 시간`
- 저장: `최고 생존 시간만 저장`
- 게임오버: `게임오버 텍스트 + 재시작 버튼`
- 연출: 눈보라 강도는 `연속값 기반`, `시간에 따라 천천히 상승`, 우선은 `체온에만 영향`, `Fog/Particle/Audio`는 보류
- 수치 튜닝: `Inspector에서 쉽게 변경 가능`

**2. 전체 커밋 분할표**
1. 프로젝트 기반 세팅, 입력 액션, 기본 폴더 구조
2. 플레이어 이동, 쿼터뷰 카메라, PC/모바일 공용 입력
3. 생존 스탯, 눈보라 강도, 게임오버 루프
4. 장작 수집, 인벤토리, 화톳불 자동 점화/연료 시스템
5. 도구 제작, 음식 자동 생성, 돈 전환 루프
6. HUD, 게임오버 UI, 최고 기록 저장
7. README 작성과 Editor 수동 작업 문서화

커밋 1 - 프로젝트 기반 세팅과 입력 자산 준비

- 전제 조건: 빈 `Unity 6.3 URP` 프로젝트가 열려 있어야 한다.
- 생성할 파일: `Assets/_Project/Scripts/Core/GameConstants.cs`, `Assets/_Project/Scripts/Core/GameTags.cs`, `Assets/_Project/Scripts/Core/GameLayers.cs`, `Assets/_Project/Input/BlizzardSurvival.inputactions`
- 수정할 파일: `Packages/manifest.json` 확인, `ProjectSettings/InputSystemPackageSettings.asset` 관련 설정 안내가 필요하면 README용 메모만 남긴다

- 주의사항: 아직 씬 자동 생성 코드는 만들지 않는다. 값 튜닝은 이후 `GameBalanceConfig`로 옮길 예정이므로 상수 클래스 남용 금지.
- 완료 조건: 입력 액션 파일과 코어 상수 파일이 준비되고, 이후 커밋에서 참조할 네이밍이 고정된다.
- 수동 테스트 방법: Unity에서 Input Actions 에셋을 열고 `Gameplay/Move`와 `Restart`가 존재하는지 확인한다.
- 추천 커밋 메시지: `chore: initialize project structure and input actions`
- Codex에 넣을 실제 프롬프트: `Unity 6.3 URP 프로젝트 기준으로 Assets/_Project 폴더 구조를 사용하는 초기 골격을 만든다. Assets/_Project/Scripts/Core 아래에 GameConstants.cs, GameTags.cs, GameLayers.cs를 생성하고, 입력 액션 파일 BlizzardSurvival.inputactions를 Assets/_Project/Input 아래에 만든다. 입력 액션에는 Gameplay 액션맵과 UI 액션맵을 두고 Gameplay/Move(Vector2), Gameplay/Restart(Button)를 정의한다. 키보드는 WASD와 Arrow 지원, Android용 가상 조이스틱 입력을 나중에 같은 Move 액션에 주입할 수 있게 설계한다. 아직 플레이어나 UI 구현은 하지 말고, 이후 커밋에서 참조할 클래스명/경로/액션명을 안정적으로 고정하는 데 집중한다.`

커밋 2 - 플레이어 이동과 쿼터뷰 카메라
- 목표: CharacterController 기반 이동과 추적 카메라를 완성한다. PC 입력과 모바일 가상 조이스틱 입력이 같은 이동 경로를 쓰도록 만든다.
- 전제 조건: 커밋 1의 입력 액션이 존재한다.
- 생성할 파일: `Assets/_Project/Scripts/Player/PlayerInputReader.cs`, `Assets/_Project/Scripts/Player/PlayerMover.cs`, `Assets/_Project/Scripts/Camera/QuarterViewCameraFollow.cs`, `Assets/_Project/Scripts/UI/VirtualJoystick.cs`
- 수정할 파일: 입력 액션에서 `Move` 바인딩 보완 필요 시 수정
- 구현 상세 단계: `PlayerInputReader`는 `MoveInput`을 외부에서 읽을 수 있게 하고, `VirtualJoystick`의 출력값도 합성할 수 있게 설계한다. `PlayerMover`는 `CharacterController`를 요구 컴포넌트로 두고 평면 이동만 처리하며 Y축 중력/점프는 넣지 않는다. 플레이어 모델의 전방이 이동 방향을 향하도록 회전시키고, 카메라는 플레이어를 따라다니는 고정 각도/고정 거리 추적형으로 만든다. `QuarterViewCameraFollow`는 `target`, `offset`, `followSmooth`, `lookAtTargetHeight`를 Inspector에서 조정 가능하게 둔다. `VirtualJoystick`은 Canvas 위에서 드래그 입력을 `Vector2`로 제공하고, PC에서는 비활성 상태여도 오류 없이 동작하게 한다.
- 주의사항: Cinemachine을 쓰지 말고 순수 스크립트로 구현한다. 모바일 입력 UI는 같은 씬에서 공존 가능해야 한다.
- 완료 조건: 키보드와 가상 조이스틱 양쪽으로 플레이어가 동일하게 이동하고, 카메라가 부드럽게 추적한다.
- 수동 테스트 방법: 플레이 모드에서 WASD 이동 확인, 조이스틱 오브젝트를 활성화한 뒤 드래그 이동 확인.
- 추천 커밋 메시지: `feat: add player movement and quarter-view camera`
- Codex에 넣을 실제 프롬프트: `Assets/_Project/Scripts/Player에 PlayerInputReader.cs와 PlayerMover.cs를 만들고, Assets/_Project/Scripts/Camera에 QuarterViewCameraFollow.cs를 만든다. Assets/_Project/Scripts/UI에 VirtualJoystick.cs를 만든다. PlayerMover는 CharacterController 기반 평면 이동만 처리하고 점프와 복잡한 경사 처리는 제외한다. PlayerInputReader는 New Input System Move 입력과 VirtualJoystick 입력을 합쳐서 CurrentMoveVector를 노출한다. QuarterViewCameraFollow는 target을 따라 고정 각도와 거리로 추적하며 LookAt 또는 등가 동작으로 플레이어 상단을 본다. VirtualJoystick은 모바일 Canvas에 배치될 전제로 포인터 다운/드래그/업을 처리해 normalized Vector2를 내보낸다. 클래스명, 공개 필드명, 이벤트 연결 포인트를 명확하게 설계하고 null-safe 하게 작성한다.`

커밋 3 - 생존 스탯, 눈보라 강도, 게임오버 상태
- 목표: 시간 경과에 따른 체온 감소, 눈보라 강도 상승, 체력 감소, 게임오버를 만든다.
- 전제 조건: 플레이어 이동과 카메라가 동작해야 한다.
- 생성할 파일: `Assets/_Project/Scripts/Config/GameBalanceConfig.cs`, `Assets/_Project/Scripts/Systems/SurvivalSystem.cs`, `Assets/_Project/Scripts/Systems/BlizzardSystem.cs`, `Assets/_Project/Scripts/Systems/GameFlowController.cs`
- 수정할 파일: `PlayerMover.cs`에서 생존 관련 의존성이 필요하면 최소한으로 연결
- 구현 상세 단계: `GameBalanceConfig`는 `ScriptableObject`로 만들고 `maxHealth=100`, `maxTemperature=100`, `baseTemperatureLossPerSecond=1`, `temperatureRecoveryPerSecond=5`, `healthLossWhenFrozenPerSecond=2`, `startingCampfireBurnSeconds=30`, `burnSecondsPerWood=3`, `toolWoodCost=5`, `foodWoodCost=5`, `foodHealAmount=1`, `moneyPerOverflowFood=10` 등을 Inspector 노출한다. `BlizzardSystem`은 게임 시작부터 `currentSeverity`가 천천히 0에서 상승하고, 최대값과 증가 속도를 Inspector에서 조정 가능하게 둔다. `SurvivalSystem`은 현재 체온/체력을 관리하고, 화톳불 범위 안이면 체온 회복, 아니면 기본 감소 + 눈보라 추가 감소를 적용한다. 체온 0이면 체력 감소. `GameFlowController`는 생존 시간 누적, 게임오버 상태 전환, 재시작 API를 제공한다.
- 주의사항: UI 의존성을 넣지 말고 이벤트 또는 프로퍼티로 상태를 노출한다.
- 완료 조건: 아무것도 하지 않아도 시간이 흐르면 눈보라 강도가 상승하고 체온/체력이 규칙대로 감소하며, 체력이 0이면 게임오버 상태가 된다.
- 수동 테스트 방법: Inspector에서 손실/회복 수치를 크게 올려 빠르게 게임오버까지 확인한다.
- 추천 커밋 메시지: `feat: implement survival, blizzard, and game flow systems`
- Codex에 넣을 실제 프롬프트: `ScriptableObject 기반 GameBalanceConfig를 도입해 대부분의 수치가 Inspector에서 조정 가능하도록 만든다. BlizzardSystem은 currentSeverity를 시간에 따라 연속적으로 증가시키고, SurvivalSystem은 현재 체온/체력 값을 관리한다. 체온은 기본 초당 1 감소에 currentSeverity 기반 추가 감소를 합산하고, 캠프파이어 열 범위 안에서는 초당 5 회복한다. 체온이 0이면 체력은 초당 2 감소한다. GameFlowController는 survivalTime 누적, isGameOver 상태, RestartGame() 메서드를 제공한다. 아직 UI는 만들지 말고, 다른 시스템이 읽을 수 있는 이벤트/속성 구조를 우선 안정적으로 만든다.`

커밋 4 - 장작 수집, 인벤토리, 화톳불 자동 점화/연료
- 목표: 월드 장작 수집과 화톳불 자동 점화/연료 소모를 만든다.
- 전제 조건: 생존 시스템과 밸런스 설정이 있어야 한다.
- 생성할 파일: `Assets/_Project/Scripts/Inventory/PlayerInventory.cs`, `Assets/_Project/Scripts/World/WoodPickup.cs`, `Assets/_Project/Scripts/World/CampfireController.cs`, `Assets/_Project/Scripts/World/CampfireHeatZone.cs`
- 수정할 파일: `SurvivalSystem.cs`, `GameTags.cs`
- 구현 상세 단계: `PlayerInventory`는 `woodCount`, `toolCount`, `foodCount`, `moneyCount`를 가진다. `WoodPickup`은 플레이어 트리거 진입 시 자동으로 장작 1개 이상을 인벤토리에 넣고 자신을 비활성 또는 파괴한다. `CampfireController`는 `isLit`, `remainingBurnTime`, `autoIgniteRadius`, `heatRadius`를 가지며, 꺼진 상태에서 플레이어가 범위에 들어오고 장작이 있으면 장작 1개를 자동 소모해 점화한다. 켜진 동안 `remainingBurnTime`이 감소하고 0 이하가 되면 꺼진다. 플레이어가 근처에 있는 동안 장작이 있으면 자동으로 연료를 계속 보충하도록 구현한다. `CampfireHeatZone`은 플레이어가 열 범위 안에 있는지 `SurvivalSystem`에 알려준다.
- 주의사항: 화톳불 3개 구조를 염두에 두되, 개수 하드코딩 금지. 자동 상호작용이 중복 발동되지 않게 debounce 처리.
- 완료 조건: 장작을 주워 인벤토리에 저장할 수 있고, 메인 화톳불과 보조 화톳불이 자동 점화/소화/연료 소모 규칙대로 동작한다.
- 수동 테스트 방법: 장작 줍기 확인, 꺼진 화톳불 근처에서 장작 자동 소모 후 점화 확인, 연료 소진 후 소화 확인.
- 추천 커밋 메시지: `feat: add wood collection and campfire automation`
- Codex에 넣을 실제 프롬프트: `PlayerInventory.cs를 만들어 장작/도구/음식/돈 카운트를 한 곳에서 관리한다. WoodPickup.cs는 Trigger 기반 자동 획득 오브젝트로 구현한다. CampfireController.cs는 화톳불 하나의 점화 상태와 남은 연료 시간을 관리하며, 플레이어가 autoIgniteRadius 안에 있고 woodCount가 1 이상이면 인벤토리 장작 1개를 자동 소모해 점화한다. 점화 후에는 burnSecondsPerWood 기준으로 남은 시간을 늘리고, 플레이어가 계속 근처에 있으면 장작을 자동으로 추가 투입해 연료를 연장한다. CampfireHeatZone.cs는 플레이어가 heatRadius 안에 있는지를 SurvivalSystem에 안전하게 전달한다. 화톳불 수는 씬 배치로 해결하고 코드상으로는 재사용 가능한 개별 컨트롤러로 작성한다.`

- 구현 상세 단계: `CraftingSystem`은 플레이어가 도구 미보유 상태에서 장작 5개 이상이 되면 자동으로 장작 5개를 소모해 도구 1개를 제작한다. 도구는 영구 보유이므로 bool 또는 count 1 이상으로 관리한다. 이후 플레이어가 화톳불 근처에 있고 도구를 보유 중이며 장작 5개 이상이면 자동으로 음식 1개를 만든다. 음식 생성 시 체력이 최대보다 낮으면 즉시 체력 +1 하고 음식 카운트도 1 증가할지, 아니면 즉시 소비만 할지 정책을 정한다. 사용자 요구는 HUD에 음식 수가 있어야 하므로 `foodCount`를 실제로 올리고, 동시에 자동 섭취 정책을 명확히 정의해야 한다. 권장 구현은 `foodCount += 1` 후 `TryAutoConsumeFood()`에서 체력이 최대 미만이면 foodCount를 1 줄이고 체력을 1 회복, 최대면 foodCount를 1 줄이고 money += moneyPerOverflowFood`로 처리한다.
- 주의사항: 무한 루프 방지. 한 프레임에 여러 번 자동 제작되지 않도록 쿨다운 또는 while 제한을 둔다.
- 완료 조건: 장작 수집만으로 도구 자동 제작이 이루어지고, 화톳불 근처에서 음식 생성 및 체력/돈 전환이 자동으로 동작한다.

- 추천 커밋 메시지: `feat: add automatic crafting, food, and money loop`
- Codex에 넣을 실제 프롬프트: `CraftingSystem.cs를 도입해 매우 단순한 자동 제작 루프를 구현한다. 플레이어가 toolCount 0이고 woodCount가 toolWoodCost 이상이면 장작 5개를 소모해 도구 1개를 자동 제작한다. 도구는 영구 보유다. 플레이어가 캠프파이어 열 범위 안에 있고 도구를 보유하며 woodCount가 foodWoodCost 이상이면 장작 5개를 소모해 음식 1개를 만든다. 음식은 HUD 표시 요구가 있으므로 inventory.foodCount를 실제로 올린 뒤 자동 소비 로직을 적용한다. 현재 체력이 최대보다 낮으면 음식 1개를 즉시 소비해 체력 1 회복, 체력이 최대면 음식 1개를 즉시 돈 10으로 전환한다. 관련 수치는 모두 GameBalanceConfig에서 수정 가능해야 한다.`
